# AGENTS.md

本文件约束在此仓库工作的自动化编码代理。适用于仓库根目录及全部子目录；若子目录出现更具体的 `AGENTS.md`，以更接近目标文件的规则为准。

## 项目定位

- 本项目是《铁砧工艺》（AnvilCraft）方块最密堆积计算器，不是物品组数或空间超压器质量配料计算器。
- 支持两类场景：虚空物质块/负物质块/虚空能收集器，以及钚块/铅块/集热器。
- 当前技术栈是 Vue 3、TypeScript、Vite 和 pnpm；界面依赖已有 Arco Design。
- 产品规则、尺寸定义、优化目标、里程碑和验收标准以 `PLAN.md` 为准。
- Minecraft 模型和纹理位于 `src/assets/anvilcraft/`，使用前阅读 `ASSETS_LICENSE`。

## 开始工作前

1. 阅读 `PLAN.md`、`README.md`、`package.json`、`ASSETS_LICENSE` 和将要修改的文件。
2. 运行 `git status --short`，保留用户已有修改；不要重置、覆盖或格式化无关工作。
3. 确认任务依赖的模组版本和衰变规则已固定。没有版本证据时，将规则保持为可配置数据并明确未确认项，不要猜测。
4. 检查现有模型、纹理和依赖后再设计接口，避免重新创建仓库已经包含的资源。
5. 保持任务范围聚焦，除非完成任务确实需要，不做无关重构或依赖升级。

## 架构边界

- `src/domain/`：纯 TypeScript 网格、坐标、规则、求解请求/结果和独立验证器。不得依赖 Vue、DOM、Three.js、网络或 `localStorage`。
- `src/data/`：版本化场景规则和资源清单。组件中不得散落设备间距、角色映射或衰变条件。
- `src/rendering/`：Minecraft JSON 到 Three.js 的解析与资源缓存。不得包含布局优化规则。
- `src/workers/`：整数求解器及消息协议。Worker 返回领域结果，不返回 Three.js 对象。
- `src/composables/`：连接 Vue 状态、Worker 生命周期、缓存和 3D/分层视图联动。
- `src/components/`：表单、统计、3D 工作区和分层预览。组件不得自行生成或修正布局。
- `src/utils/`：无领域含义的格式化、复制等小工具。不要把坐标或求解规则藏进 `utils`。
- `App.vue` 只负责页面组合和顶层状态，不承载求解器或模型解析器。

优先使用 props/emits 和组合式函数。只有出现真实的跨页面或复杂共享状态需求时才引入路由或状态管理库。

## 坐标与结构规则

- 所有领域坐标均为从 `0` 开始的整数 `(x, y, z)`；坐标轴含义不能在求解器和视图间交换。
- 一个基础单元固定为 `5 × 5 × 5`。设备局部坐标为 `(2, 2, 2)`。
- 总设备坐标为 `(5x + 2, 5y + 2, 5z + 2)`，每个基础单元恰好一台设备。
- 不堆积模式固定为一个 `1 × 1 × 1` 单元和一台设备；单元内部仍需求解最少隔离材料。
- 平面模式的单元尺寸是 `m × 1 × n`；立体模式是 `l × m × n`。
- 模式类型固定为 `single | planar | volumetric`。`single` 请求必须规范化为 `units = (1,1,1)`，不能受隐藏的旧尺寸字段影响。
- 相邻单元组成一个连续网格，必须联合求解。禁止逐个 `5 × 5 × 5` 求解后拼接。
- 六面邻接只包含 `±X`、`±Y`、`±Z`，不包含边和角。
- 当前两种场景均在五个或更多正交邻居为同类主材料时衰变，因此每个主材料格至少要有两个方向不是主材料；结构外和设备都视为非主材料。
- 领域角色固定为 `primary`、`separator`、`device`。具体方块名称和资源由场景数据映射。
- 坐标线性化和反线性化只能有一个实现来源，并被求解、验证、3D 和分层视图共用。

## 最优性与求解器

- 主目标是隔离材料数量的全局最小值：负物质块或铅块越少越好。
- “最密/最优”只能用于已证明最优的结果。超时得到的可行解必须显式标为 `feasible` 并展示上下界。
- 使用成熟的浏览器可用 0-1/MILP 求解器并在 Web Worker 中运行；不要用贪心、局部搜索或手写启发式冒充精确最优。
- 每个可放置格使用二元变量。设备位置固定，不得被求解器覆盖。
- 同分结果使用 `PLAN.md` 定义的稳定次级顺序；不得依赖随机种子、对象键顺序或不稳定排序。
- 每个求解结果在进入 UI 前必须由独立验证器复核设备位置、位置总数和逐格六面约束。
- 验证器不得复用求解器内部约束对象，以免同一个建模错误同时污染生成和验证。
- 设置明确的尺寸和时间上限；取消旧请求，不能让旧 Worker 结果覆盖新输入。
- 小规模穷举参考求解器仅用于测试预言机，不直接处理生产尺寸。
- 当前五同类邻面规则下，基础 `5 × 5 × 5` 回归基线是 110 个主材料、14 个隔离材料和 1 台中心设备；不得把它当成其他游戏版本的常量。

## 场景数据与游戏版本

- 场景数据必须包含模组版本、Minecraft 版本、规则来源修订、核对日期和衰变规则 ID。
- 虚空场景映射为虚空物质块、负物质块和虚空能收集器。
- 钚场景映射为钚块、铅块和集热器；当前规则与虚空物质相同，但跨版本更新时必须分别复核。
- 运行时只读取站点随包发布的本地规则和资源，不依赖远程 API。
- 规则或版本变化时先更新数据、`PLAN.md` 和黄金测试，再修改界面表现。

## Three.js 与 Minecraft 资源

- 3D 必须使用 Three.js，并复用 `src/assets/anvilcraft/` 中的模型和纹理，不用同色占位立方体替代最终效果。
- 模型为 Minecraft/Blockbench JSON，不是 glTF。使用结构化解析器支持当前资源实际出现的字段子集，遇到不支持内容应明确报错。
- 必须正确解析 `minecraft:block/block`、`minecraft:block/cube_all`、texture variables、elements、逐面 UV/rotation、反向盒、透明/裁切和发光字段。
- 虚空能收集器和集热器由 base/head 模型组成；保持各自局部原点和纹理，不要错误合并为普通整方块。
- 纹理使用最近邻采样和正确色彩空间；透明材质不能产生整块黑面。
- 大量重复方块使用 `InstancedMesh` 或等价批处理。不要为每个格子复制纹理、材质或创建独立动画循环。
- 共享并缓存 geometry/material/texture，组件卸载和结构切换时正确释放 GPU 资源。
- Three.js 场景是主要工作区，应无装饰性卡片边框；自动取景必须适配实际结构长宽高。
- 3D 显隐/裁切和二维分层只能筛选同一份 `StructureResult`，不能重新计算方块类型。
- 修改、增删素材时同步检查并更新 `ASSETS_LICENSE`，不要删除模型内原作者 credit。

## Vue 与 TypeScript

- 使用 Composition API 和 `<script setup lang="ts">`。
- 为领域边界、Worker 消息、组件 props/emits、模型 JSON 和持久化数据提供明确类型；避免 `any` 和无依据断言。
- 派生状态优先用 `computed`，不要用 watcher 维护可直接推导的重复状态。
- watcher、Worker 和资源加载必须识别过期请求并清理副作用。
- 用户输入保留适合编辑的表单状态，在单一入口规范化；不让 `NaN` 或非法尺寸进入领域层。
- 组件保持小而聚焦，只在确实减少重复或隔离复杂性时新增抽象。
- 保持严格 TypeScript 检查，不关闭规则掩盖错误。

## 界面要求

- 首屏直接呈现计算器，桌面端让 3D 场景成为主要视觉和交互区域，不制作营销页。
- 风格紧凑、安静、适合反复调参；避免巨型标题、卡片套卡片和无关装饰。
- 使用现有 Arco Design 控件和图标。不堆积/平面/立体模式及轴向用分段控件，布尔显示设置用开关或复选框，层号和尺寸用合适的数字输入/滑块。
- 不堆积模式隐藏尺寸输入并明确显示固定尺寸 `5 × 5 × 5`；切换模式时清理不再适用的字段。
- 只有图标的按钮必须有可访问名称和 tooltip。
- 3D 提供旋转、缩放、平移、重置、角色显隐和轴向裁切；分层视图提供 X/Y/Z、完整层范围和当前层统计。
- 主材料、隔离材料和设备不能只靠颜色区分，图例、纹理或图案至少提供一种额外信号。
- 求解中、最优、仅可行、无解、失败是不同状态，并让读屏软件感知动态变化。
- 所有核心操作可用键盘完成，焦点清晰，触控目标合理。
- 在 360、768、1024、1440 px 宽度检查文本、画布、工具栏和分层网格，不得重叠或越界。
- 不使用随视口连续缩放的字体，不使用负字距。

## 测试要求

- 每次修复求解器、坐标或模型解析缺陷都添加先前会失败的回归测试。
- 领域测试覆盖三种模式的尺寸规范化、设备坐标、边界、恰好五个同类邻面的衰变约束、位置守恒、联合求解和确定性。
- 组件测试覆盖不堆积模式的固定尺寸、尺寸输入隐藏及模式切换后的旧值清理。
- 小规模随机案例将生产求解器与穷举参考求解器比较最优隔离材料数量。
- 求解器与独立验证器分开测试；不能只测试求解器自报 `optimal`。
- 模型解析测试覆盖 cube parent、复杂 elements、UV rotation、反向盒、透明、发光和 base/head 组合。
- 组件测试覆盖实际输入、求解状态、角色显隐、裁切、轴向切层、层边界、联动和复制行为。
- 3D 完成后必须用浏览器截图和 canvas 像素检查验证桌面/移动端非空、自动取景正确、资源可见且无重叠。
- 性能测试记录求解耗时、Worker 取消、峰值内存、3D 首帧和交互表现。
- 避免用大量脆弱快照代替行为和数值断言。

一次改动只有在以下条件满足时才算完成：

- 行为符合 `PLAN.md` 和本次任务验收标准。
- 领域规则和坐标约定只有一个实现来源，游戏版本清晰。
- 正常路径、边界、失败路径及相关最优性有测试。
- 所有结果通过独立布局验证器。
- TypeScript 检查、相关测试和生产构建通过。
- 3D 与分层预览逐格一致，桌面/移动视口不存在空白、重叠、溢出或不可操作状态。
- 没有破坏用户已有修改，也没有夹带无关重构或生成文件。
