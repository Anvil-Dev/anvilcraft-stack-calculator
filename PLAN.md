# 铁砧工艺最密堆积计算器实施计划

## 1. 项目目标

制作一个无需后端、可直接在浏览器中使用的《铁砧工艺》（AnvilCraft）方块最密堆积计算器。计算器针对两类结构：

- 虚空发电：使用虚空物质块和负物质块，正中心放置虚空能收集器。虚空物质块的六个面如果都与虚空物质块相邻就会衰变，因此必须混入昂贵的负物质块，并使其数量达到可证明的最小值。
- 钚块集热：使用钚块和铅块，正中心放置集热器。铅块用于打断会导致钚块衰变的邻接结构，并应尽可能少用。

用户选择结构类型和堆积模式，并在平面/立体模式下输入单元数量后，应用生成一个有效且最优的三维布局，同时提供可交互的 3D 预览、逐层预览、材料统计和可执行的建造信息。

当前仓库是 Vue 3、TypeScript 6 和 Vite 8 的初始模板。`src/assets/anvilcraft/` 已包含两类结构所需的 Minecraft 方块模型和纹理。

## 2. 术语与坐标

### 2.1 单元与尺寸

一个基础发电/集热单元占用 `5 × 5 × 5` 个方块位置，其中正中心是设备，其余位置由主材料或隔离材料填充。

- 不堆积：固定为一个独立的 `5 × 5 × 5` 基础单元，即三个方向的单元数均为 `1`。
- 平面最密堆积：`5m × 5 × 5n`，即 X、Y、Z 方向分别有 `m`、`1`、`n` 个基础单元。
- 立体最密堆积：`5l × 5m × 5n`，即 X、Y、Z 方向分别有 `l`、`m`、`n` 个基础单元。
- `l`、`m`、`n` 均为正整数。

“不堆积”只表示不与其他基础单元相连；单元内部仍需求解最少隔离材料，并不是跳过优化。

实现统一使用从 `0` 开始的整数坐标。若三个方向的单元数为 `unitsX`、`unitsY`、`unitsZ`：

```text
blocksX = 5 * unitsX
blocksY = 5 * unitsY
blocksZ = 5 * unitsZ

设备坐标 = (5x + 2, 5y + 2, 5z + 2)
其中 0 <= x < unitsX, 0 <= y < unitsY, 0 <= z < unitsZ
```

因此不堆积模式固定有 1 个设备，平面模式有 `m * n` 个设备，立体模式有 `l * m * n` 个设备。平面和立体模式中的相邻基础单元共享连续边界，必须在整个结构上联合求解；禁止先分别优化每个 `5 × 5 × 5` 单元再简单拼接，因为单元交界处的邻接会改变最优解。

### 2.2 方块角色

在两种场景都采用“六面同类主材料包围才衰变”规则的前提下，它们共享同一拓扑模型，但使用不同方块和设备资源：

| 场景 | 主材料 | 需要最小化的隔离材料 | 中心设备 |
| --- | --- | --- | --- |
| 虚空发电 | 虚空物质块 | 负物质块 | 虚空能收集器 |
| 钚块集热 | 钚块 | 铅块 | 集热器 |

领域层只处理 `primary`、`separator`、`device` 三种角色，具体名称、模型和纹理由版本化场景数据映射，求解器不得依赖素材文件名。

### 2.3 六面邻接规则

只计算共享完整方块面的六个正交邻居：`±X`、`±Y`、`±Z`。边、角接触不算相邻。

对结构内每个主材料格，以下条件必须成立：

- 六个正交邻居中至少有一个不是主材料；它可以是隔离材料、中心设备或结构外部空间。
- 结构边界外视为非主材料，因此边界上的主材料天然不会被六面完全包围。
- 中心设备是固定的非主材料位置，不计入主材料或隔离材料数量。

虚空物质块按上述规则处理。钚块是否在目标模组版本中使用完全相同的“六面均为钚块才衰变”判定，必须在 M0 用实际版本源码或游戏测试确认；求解器应支持把衰变规则配置为场景数据，不能把未经确认的版本行为写死在 UI 中。

## 3. 优化目标与正确性

主目标只有一个：在所有满足衰变约束的完整布局中，最小化隔离材料数量。

- 虚空发电最小化负物质块数量，同时最大化虚空物质块数量。
- 钚块集热最小化铅块数量，同时最大化钚块数量。
- 总方块位置和设备位置固定，因此两种表述等价。

若存在多个隔离材料数量相同的最优布局，使用稳定的次级排序获得确定结果：

1. 优先重复层/周期更少、便于实际建造的方案；此指标必须在 M0 给出精确定义。
2. 再按固定坐标顺序比较隔离材料位置，保证相同输入产生相同布局。

界面只有在求解器证明达到最优下界时才能显示“最密/最优”。若达到时间限制但只找到可行解，必须显示“可行方案”和当前上下界差距，不能将其冒充最优解。MVP 可以通过限制最大 `l/m/n`，确保支持范围内都能证明最优。

## 4. M0 必须确认的规则

实现求解器前需要固定并记录：

- 目标《铁砧工艺》版本及 Minecraft/NeoForge 版本。
- 钚块衰变的精确邻接条件，及铅块是否只作为非钚邻居使用。
- 每个连续 `5 × 5 × 5` 单元是否都必须在局部中心放置一台设备；本计划按“是”处理。
- 结构外部是否始终视为安全的非主材料；本计划按“是”处理。
- 两个相邻基础单元间是否没有空隙；本计划按连续体积处理。
- 平面模式是否固定 Y 方向只有一个单元，以及界面中 `m/n` 与游戏方向的对应关系。
- 最优解同分时“易建造性”的精确指标；未确认前只使用固定坐标顺序。
- 浏览器端允许的最大单元数和可接受求解时间。

这些结论要写入场景数据元信息和黄金测试，不能只保存在聊天记录中。

## 5. MVP 功能范围

### 5.1 输入

- 场景选择：虚空发电、钚块集热。
- 堆积模式：不堆积、平面、立体。
- 不堆积模式不显示尺寸输入，并将单元尺寸规范化为 `1 × 1 × 1`。
- 平面模式输入 `m`、`n`；立体模式输入 `l`、`m`、`n`。
- 所有单元数必须为受产品上限约束的正整数，并提供空值、零、负数、小数和超限提示。
- 模式切换必须清除不适用于新模式的旧尺寸，不允许隐藏字段继续影响请求。
- 一键恢复为不堆积的 `1 × 1 × 1` 基础单元。

### 5.2 结果

- 求解状态：最优、可行但未证明最优、无解、输入无效或求解失败。
- 结构总尺寸、基础单元数、设备数、主材料数、隔离材料数、总方块数。
- 隔离材料占比，以及求解器的最优性状态/上下界。
- 可复制的材料清单和隔离材料坐标清单。
- 3D 预览和分层预览必须使用同一个不可变计算结果，不能分别生成布局。

### 5.3 3D 预览

- 使用 Three.js 渲染真实模型和纹理，场景作为主要工作区，不放在装饰性预览卡片中。
- 支持旋转、缩放、平移、自动居中和重置视角。
- 支持显示/隐藏主材料、隔离材料和设备，或降低主材料透明度以检查内部布局。
- 支持按 X/Y/Z 轴裁切可见范围，定位内部错误或建造层。
- 提供图例、当前尺寸和明确的加载/失败/空状态。
- 大结构使用实例化或等价批处理，不能为每个方块创建独立材质和纹理副本。

### 5.4 分层预览

- 使用 X/Y/Z 分段控件选择切片轴。
- 使用滑块、步进器和前后层按钮选择层索引，显示人类可读层号与内部坐标。
- 用稳定尺寸的二维方格展示当前层；主材料、隔离材料和设备同时使用纹理/图案与文字图例区分，不能只依赖颜色。
- 显示当前层三类方块数量，并可复制该层隔离材料坐标。
- 3D 裁切层与分层预览的轴和层索引应能联动。

### 5.5 MVP 不包含

- 在线账号、云同步、排行榜或后端数据库。
- 自动读取游戏存档或玩家库存。
- 在浏览器运行时从 Wiki/GitHub 抓取规则或素材。
- 自动在游戏世界中放置结构、生成 schematic/Litematica 文件。
- 用户手工编辑单个格子；首版只展示经过求解和验证的布局。
- 未经证明最优的大尺寸近似方案被标记为“最密”。

## 6. 领域与数据模型

建议目录：

```text
src/
  assets/anvilcraft/   # 已授权的 Minecraft 模型和纹理源文件
  components/          # 表单、统计、3D 工作区、分层预览
  composables/         # 页面状态、Worker 生命周期、视图联动
  data/                # 版本化场景规则和资源清单
  domain/              # 网格、约束、求解请求/结果、独立验证器
  rendering/           # Minecraft 模型解析与 Three.js 适配
  workers/             # 求解器 Worker 入口
  utils/               # 无领域含义的格式化和复制工具
  App.vue               # 页面组合，不承载求解或渲染细节
```

核心类型至少覆盖：

```ts
type ScenarioId = 'void-energy' | 'plutonium-heat'
type StackMode = 'single' | 'planar' | 'volumetric'
type CellType = 'primary' | 'separator' | 'device'

interface ModuleDimensions {
  x: number
  y: number
  z: number
}

interface StructureRequest {
  scenario: ScenarioId
  mode: StackMode
  units: ModuleDimensions
}

interface Cell {
  x: number
  y: number
  z: number
  type: CellType
}

interface SolverMetadata {
  status: 'optimal' | 'feasible' | 'infeasible' | 'failed'
  lowerBound: number
  upperBound: number
  durationMs: number
}

interface StructureResult {
  request: StructureRequest
  blocks: ModuleDimensions
  cells: readonly Cell[]
  primaryCount: number
  separatorCount: number
  deviceCount: number
  solver: SolverMetadata
}
```

生产实现可将密集 `Cell[]` 替换为类型化数组或按坐标计算的紧凑表示，但对外语义必须一致。坐标到线性索引的顺序只能定义一次，并由求解器、验证器、3D 和分层视图共同使用。

场景数据至少包含：模组版本、Minecraft 版本、规则来源修订、核对日期、三种角色的显示名称、模型路径、纹理路径和衰变规则 ID。

## 7. 求解器设计

### 7.1 数学模型

为每个可放置材料的位置 `p` 定义二元变量：

```text
x[p] = 1  表示隔离材料
x[p] = 0  表示主材料
```

设备位置是固定洞，不创建变量。对于既不在结构边界、也不与设备相邻的每个位置 `p`，加入约束：

```text
x[p] + Σ x[q] >= 1
q 为 p 的六个正交邻居
```

该约束表示：`p` 自身是隔离材料，或它作为主材料时至少有一个隔离材料邻居。边界或设备邻接已经提供固定的非主材料邻居，因此不需要这条约束。

目标函数：

```text
minimize Σ x[p]
```

这是带固定设备洞的三维网格最小支配集/0-1 整数优化问题。实现应优先评估成熟的浏览器可用 MILP 求解器（例如 HiGHS 的 WASM 构建或等价方案），核对许可证、包体、确定性和 Worker 支持；不要手写未经充分验证的启发式算法替代精确求解器。

按当前六面规则，`1 × 1 × 1` 基础单元的初始拓扑黄金用例为：

```text
总尺寸：5 × 5 × 5
设备：(2, 2, 2)
最少隔离材料：6
主材料：118
隔离材料：6
设备：1

一组可行的隔离材料坐标：
(2,1,1), (2,1,3), (2,3,1), (2,3,3), (1,2,2), (3,2,2)
```

该最小值已通过对 20 个潜在衰变位置的穷举集合覆盖验证，坐标布局也通过逐格六面检查；它仍需在 M0 用目标游戏版本确认衰变规则后，才能成为正式游戏规则黄金用例。

### 7.2 求解流程

1. 校验模式与单元数，并生成总尺寸和全部固定设备坐标。
2. 建立连续网格的稀疏约束矩阵，而不是逐个基础单元求解。
3. 使用已知周期构造或上一次结果作为可行上界，用简单计数界或求解器松弛作为下界。
4. 在 Web Worker 中运行求解器，支持超时、取消和请求序号，防止旧结果覆盖新输入。
5. 只接受整数解；按固定坐标顺序做确定性同分处理。
6. 将求解结果交给独立验证器，逐格检查设备位置、方块总数和所有六面衰变约束。
7. 验证通过后生成统计、3D 实例数据和分层索引；验证失败则显示内部错误，不渲染错误结构。

### 7.3 性能边界

- M0 通过基准测试确定 `l/m/n` 上限，不凭感觉承诺任意尺寸。
- 求解期间主线程应保持可交互，连续输入只保留最新请求。
- 已完成的请求可按规范化参数缓存，但缓存必须包含规则/数据版本。
- 3D 渲染按模型元素和材质批量实例化；分层视图只渲染当前二维切片。
- 性能验收同时关注求解耗时、峰值内存、首帧时间和相机交互帧率。

## 8. Minecraft 资源与 Three.js

现有资源位于 `src/assets/anvilcraft/`：

- 方块：`void_matter_block`、`negative_matter_block`、`plutonium_block`、`lead_block`。
- 设备：`void_energy_collector_base/head`、`heat_collector_base/head`。
- 模型是 Minecraft/Blockbench JSON，设备由 base 与 head 两部分组合；纹理是像素 PNG。

渲染适配层需要：

- 使用结构化 JSON 解析模型，不用字符串替换。
- 解析当前资源实际使用的 `parent`、`textures`、`elements`、`from/to`、逐面 UV、UV rotation、element rotation、`shade`、`light_emission`、`ambientocclusion` 和 `render_type` 子集。
- 内置解析 `minecraft:block/block` 与 `minecraft:block/cube_all`，遇到不支持的父模型或字段时明确失败。
- 正确处理反向 `from/to` 的内部面、透明/裁切纹理、发光轮廓和 base/head 的局部原点，不能把所有模型退化成同色立方体。
- 纹理使用最近邻采样、正确色彩空间和 alpha 规则，保持 Minecraft 像素观感。
- 缓存几何体、纹理和材质，并在组件卸载或场景重建时释放 GPU 资源。

模型和纹理的版权/许可证遵循仓库中的 `ASSETS_LICENSE`，修改、增删资源时同步更新来源说明。

## 9. 页面结构

桌面端建议布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 铁砧工艺 · 最密堆积                  规则版本 / 求解状态      │
├───────────────┬──────────────────────────────────────────────┤
│ 场景与尺寸    │                                              │
│ 虚空 / 钚块   │              3D 交互工作区                   │
│ 单个/平面/立体│                                              │
│ 条件尺寸输入  │              图例与视图工具                  │
│ [重新计算]    │                                              │
├───────────────┼───────────────────────┬──────────────────────┤
│ 材料统计      │ 分层预览              │ 当前层统计/坐标      │
│ 主/隔离/设备  │ 轴向、层号、二维网格  │ 复制清单             │

```

- 计算器直接作为首屏，不制作营销页。
- 桌面端以 3D 场景为主工作区；窄屏按“参数 -> 统计 -> 3D -> 分层”排列。
- 使用现有 Arco Design 控件保持输入一致性；图标使用同一组件库提供的图标。
- 3D 工具使用图标按钮并提供可访问名称和 tooltip；模式和轴向使用分段控件。
- 所有求解状态、图例和错误不能只靠颜色表达。

## 10. 测试策略

### 10.1 领域与求解器

- 尺寸映射：不堆积 `1 × 1 × 1`、平面 `m × 1 × n`、立体 `l × m × n` 及对应的总方块尺寸。
- 不堆积模式忽略/拒绝外部传入的非 `1 × 1 × 1` 单元尺寸，避免隐藏参数污染结果。
- 设备坐标、设备数量、坐标线性化/反线性化。
- 边界主材料安全、设备邻接主材料安全、六面主材料包围必定失败。
- 隔离材料打断六面包围后通过验证。
- 总位置守恒：`primary + separator + device = blocksX * blocksY * blocksZ`。
- 两种场景在使用同一规则时拓扑结果一致，但角色/资源映射不同。
- 基础 `5 × 5 × 5` 黄金用例为 118 个主材料、6 个隔离材料和 1 台设备，并验证文档给出的坐标。
- 对小网格使用穷举参考求解器，与生产 MILP 结果比较隔离材料最小值。
- 相同请求重复运行产生相同的隔离材料坐标。
- 超时可行解不得被标记为最优，验证失败的解不得进入视图。
- 多单元联合求解用例必须覆盖单元交界，防止退化为逐单元拼接。

### 10.2 模型与渲染

- 为模型 JSON 解析器建立 fixture 测试，覆盖 `cube_all`、复杂 elements、反向盒、UV rotation、透明和发光字段。
- 验证所有场景资源路径存在、纹理可加载，base/head 能组成完整设备。
- 3D 与分层预览对同一坐标的方块类型一致。
- 更换请求后旧几何体、材质和动画循环正确清理。
- 使用 canvas 像素检查确认场景非空，并在桌面/移动视口截图验证自动取景、无重叠和内部裁切。

### 10.3 组件与端到端

- 场景切换、不堆积/平面/立体切换、条件尺寸显隐、尺寸清理、计算取消和结果更新。
- 3D 旋转/缩放/重置、角色显隐、裁切范围。
- X/Y/Z 切层、层号边界、前后层按钮、3D/2D 联动和复制坐标。
- 360、768、1024、1440 px 宽度下无横向溢出、遮挡或不可操作控件。
- 仅键盘可以完成设置尺寸、计算、切层和复制材料清单。

## 12. MVP 验收标准

- 用户可以选择虚空发电或钚块集热，并选择不堆积、平面或立体模式。
- 不堆积模式固定生成一个 `5 × 5 × 5` 单元和一台中心设备，不要求用户输入尺寸。
- 每个基础单元中心都有且只有对应设备，整体尺寸和设备数量正确。
- 每个主材料格都通过六面邻接验证，不会出现已知衰变布局。
- 隔离材料数量在支持范围内被证明最少；未证明最优时界面不声称“最密”。
- 统计、3D 场景、分层网格和复制坐标来自同一份结果，逐格一致。
- 3D 预览使用仓库中的真实模型/纹理，可旋转、缩放、显隐和检查内部层。
- 分层预览支持三个轴、完整层范围、当前层统计和设备辨识。
- 常用尺寸下求解不阻塞主线程，3D 操作保持可用。
- 360 至 1440 px 宽度下核心流程无重叠、溢出或不可操作状态。
- `pnpm build`、测试和类型检查全部通过。
