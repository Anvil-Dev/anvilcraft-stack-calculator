# 铁砧工艺最密堆积计算器实施计划

## 1. 项目目标

制作一个无需后端、可直接在浏览器中使用的《铁砧工艺》（AnvilCraft）方块最密堆积计算器。计算器针对两类结构：

- 虚空发电：使用虚空物质块和负物质块，正中心放置虚空能收集器。虚空物质块有五个或更多面与虚空物质块相邻时就会衰变，因此必须混入昂贵的负物质块，并使其数量达到可证明的最小值。
- 钚块集热：使用钚块和铅块，正中心放置集热器。每个钚块六面直接相邻的钚块数减铅块数必须不大于 `2`，集热器不参与该差值；铅块应尽可能少用。

用户选择结构类型和堆积模式后，应用生成一个有效的 `5 × 5 × 5` 单元布局；只有上下界一致时才将其标记为最优。同时提供可交互的 3D 预览、逐层预览、每单元材料统计和可执行的复制方向信息。

当前仓库是 Vue 3、TypeScript 6 和 Vite 8 的初始模板。`src/assets/anvilcraft/` 已包含两类结构所需的 Minecraft 方块模型和纹理。

## 2. 术语与坐标

### 2.1 单元与尺寸

一个基础发电/集热单元占用 `5 × 5 × 5` 个方块位置，其中正中心 `(2,2,2)` 是设备，其余位置由主材料或隔离材料填充。三种模式都只求解并展示这一份单元，不输入或保存堆积数量：

- 不堆积：单元六个外表面之外均视为空气，不与其他单元连接。
- 平面最密堆积：单元可沿 X/Z 任意复制；X/Z 两侧使用周期边界，Y 两侧仍为空气。
- 立体最密堆积：单元可沿 X/Y/Z 任意复制；三轴两侧都使用周期边界。

实现统一使用从 `0` 开始的单元局部整数坐标。所有模式的请求均规范化为 `units = (1,1,1)`，结果尺寸固定为 `blocks = (5,5,5)`，每单元恰好一台设备。平面或立体复制后的世界坐标通过各轴对 `5` 取模映射回同一份单元布局，因此每个单元的摆放完全一致。

### 2.2 方块角色

两种场景使用相同的方块角色和坐标模型，但采用不同的衰变约束：

| 场景 | 主材料 | 需要最小化的隔离材料 | 中心设备 |
| --- | --- | --- | --- |
| 虚空发电 | 虚空物质块 | 负物质块 | 虚空能收集器 |
| 钚块集热 | 钚块 | 铅块 | 集热器 |

领域层只处理 `primary`、`separator`、`device` 三种角色，具体名称、模型和纹理由版本化场景数据映射，求解器不得依赖素材文件名。

### 2.3 六面邻接规则

只计算共享完整方块面的六个正交邻居：`±X`、`±Y`、`±Z`。边、角接触不算相邻。

对结构内每个主材料格，按场景应用以下条件：

- 虚空场景：相邻虚空物质块不超过四个。负物质块、中心设备或非周期边界外都视为非虚空物质块。
- 钚场景：相邻钚块数减相邻铅块数不大于 `2`。中心集热器和非周期边界外不计入钚块数，也不计入铅块数。
- 不堆积模式的结构边界外视为非主材料。平面模式仅 Y 边界外视为非主材料，X/Z 越界邻居回绕到单元另一侧。立体模式所有越界邻居都回绕到单元另一侧。
- 中心设备是固定的非主材料位置，不计入主材料或隔离材料数量。

衰变规则作为版本化场景数据记录；切换目标游戏版本时必须分别复核，不能把当前版本行为写死在 UI 中。

## 3. 优化目标与正确性

主目标只有一个：在所有满足衰变约束的完整布局中，最小化隔离材料数量。

- 虚空发电最小化负物质块数量，同时最大化虚空物质块数量。
- 钚块集热最小化铅块数量，同时最大化钚块数量。
- 总方块位置和设备位置固定，因此两种表述等价。

若存在多个隔离材料数量相同的最优布局，使用稳定的次级排序获得确定结果：

1. 优先重复层/周期更少、便于实际建造的方案；此指标必须在 M0 给出精确定义。
2. 再按固定坐标顺序比较隔离材料位置，保证相同输入产生相同布局。

界面只有在求解器证明达到版本化最优下界时才能显示“最密/最优”。若达到时间限制但只找到可行解，必须显示“可行方案”和当前上下界差距，不能将其冒充最优解。

## 4. M0 必须确认的规则

实现求解器前需要固定并记录：

- 目标《铁砧工艺》版本及 Minecraft/NeoForge 版本。
- 两种场景的衰变规则 ID、规则来源修订和目标版本是否一致。
- 每个 `5 × 5 × 5` 单元是否都必须在局部中心放置一台设备；本计划按“是”处理。
- 不堆积、平面和立体模式各自的周期轴是否与目标游戏结构一致。
- 相邻复制单元间是否没有空隙；本计划按连续周期边界处理。
- 最优解同分时“易建造性”的精确指标；未确认前只使用固定坐标顺序。
- 浏览器端可接受的单元求解时间。

这些结论要写入场景数据元信息和黄金测试，不能只保存在聊天记录中。

## 5. MVP 功能范围

### 5.1 输入

- 场景选择：虚空发电、钚块集热。
- 堆积模式：不堆积、平面、立体。
- 三种模式都不提供堆积数量输入，单元尺寸固定显示为 `5 × 5 × 5`。
- 平面模式显示复制方向 X/Z，立体模式显示复制方向 X/Y/Z，不堆积显示“不复制”。
- 所有请求统一规范化为 `units = (1,1,1)`，不允许隐藏或历史数量字段影响请求。
- 一键恢复为不堆积模式。

### 5.2 结果

- 求解状态：最优、可行但未证明最优、无解、输入无效或求解失败。
- 单元尺寸、复制方向、每单元设备数、主材料数、隔离材料数和总方块数。
- 隔离材料占比，以及求解器的最优性状态/上下界。
- 可复制的材料清单和隔离材料坐标清单。
- 3D 预览和分层预览必须使用同一个不可变计算结果，不能分别生成布局。

### 5.3 3D 预览

- 使用 Three.js 渲染真实模型和纹理，场景作为主要工作区，不放在装饰性预览卡片中。
- 支持旋转、缩放、平移、自动居中和重置视角。
- 收集器 head 围绕固定局部轴心做 X/Y/Z 三轴旋转，不产生上下浮动。
- 支持显示/隐藏主材料、隔离材料和设备，或降低主材料透明度以检查内部布局。
- 支持按 X/Y/Z 轴裁切可见范围，定位内部错误或建造层。
- 提供图例、当前尺寸和明确的加载/失败/空状态。
- 大结构使用实例化或等价批处理，不能为每个方块创建独立材质和纹理副本。

### 5.4 分层预览

- 使用 X/Y/Z 分段控件选择切片轴。
- 使用滑块、步进器和前后层按钮选择层索引，显示人类可读层号与内部坐标。
- 用稳定尺寸的二维方格展示当前层；主材料、隔离材料和设备同时使用纹理/图案与文字图例区分，不能只依赖颜色。
- 显示当前层三类方块数量，并可复制该层隔离材料坐标。
- 3D 裁切层与分层预览的轴和层索引应能联动。

### 5.5 MVP 不包含

- 在线账号、云同步、排行榜或后端数据库。
- 自动读取游戏存档或玩家库存。
- 在浏览器运行时从 Wiki/GitHub 抓取规则或素材。
- 自动在游戏世界中放置结构、生成 schematic/Litematica 文件。
- 用户手工编辑单个格子；首版只展示经过求解和验证的布局。
- 未经证明最优的大尺寸近似方案被标记为“最密”。

## 6. 领域与数据模型

建议目录：

```text
src/
  assets/anvilcraft/   # 已授权的 Minecraft 模型和纹理源文件
  components/          # 表单、统计、3D 工作区、分层预览
  composables/         # 页面状态、Worker 生命周期、视图联动
  data/                # 版本化场景规则和资源清单
  domain/              # 网格、约束、求解请求/结果、独立验证器
  rendering/           # Minecraft 模型解析与 Three.js 适配
  workers/             # 求解器 Worker 入口
  utils/               # 无领域含义的格式化和复制工具
  App.vue               # 页面组合，不承载求解或渲染细节
```

核心类型至少覆盖：

```ts
type ScenarioId = 'void-energy' | 'plutonium-heat'
type StackMode = 'single' | 'planar' | 'volumetric'
type CellType = 'primary' | 'separator' | 'device'

interface ModuleDimensions {
  x: number
  y: number
  z: number
}

interface StructureRequest {
  scenario: ScenarioId
  mode: StackMode
  units: ModuleDimensions // 始终规范化为 (1,1,1)
}

interface Cell {
  x: number
  y: number
  z: number
  type: CellType
}

interface SolverMetadata {
  status: 'optimal' | 'feasible' | 'infeasible' | 'failed'
  lowerBound: number
  upperBound: number
  durationMs: number
}

interface StructureResult {
  request: StructureRequest
  blocks: ModuleDimensions
  cells: readonly Cell[]
  primaryCount: number
  separatorCount: number
  deviceCount: number
  solver: SolverMetadata
}
```

生产实现可将密集 `Cell[]` 替换为类型化数组或按坐标计算的紧凑表示，但对外语义必须一致。坐标到线性索引的顺序只能定义一次，并由求解器、验证器、3D 和分层视图共同使用。

场景数据至少包含：模组版本、Minecraft 版本、规则来源修订、核对日期、三种角色的显示名称、模型路径、纹理路径、衰变规则 ID，以及三种模式的已证明下界、已知上界、预计算坐标和最优性状态。

## 7. 求解器设计

### 7.1 数学模型

为每个可放置材料的位置 `p` 定义二元变量：

```text
x[p] = 1  表示隔离材料
x[p] = 0  表示主材料
```

设备位置是固定洞，不创建变量。先按模式解析每个正交邻居：周期轴越界时回绕到单元另一侧，非周期轴越界时视为结构外。

虚空场景对每个材料位置 `p`，令 `fixedSafeNeighbors` 表示属于设备或非周期结构外的位置数，加入约束：

```text
2x[p] + Σ x[q] >= 2 - fixedSafeNeighbors
q 为 p 解析回绕后的非设备正交邻居
```

该约束表示：`p` 自身是负物质块时约束自动满足；它作为虚空物质块时，负物质块邻居、设备和结构外方向合计必须至少有两个。右侧不大于零的约束可以省略。

钚场景只把相邻钚块或铅块计为材料邻居。令 `N[p]` 为 `p` 的材料邻居数，`k[p] = max(0, ceil((N[p] - 2) / 2))`，加入约束：

```text
k[p] x[p] + Σ x[q] >= k[p]
q 为 p 解析回绕后的非设备材料邻居
```

当 `p` 是钚块时，该约束要求至少 `k[p]` 个铅块邻居，等价于“相邻钚块数减相邻铅块数不大于 2”；当 `p` 是铅块时约束自动满足。集热器和结构外不进入 `N[p]`。

目标函数：

```text
minimize Σ x[p]
```

这是带固定设备洞的三维网格最小支配集/0-1 整数优化问题。实现应优先评估成熟的浏览器可用 MILP 求解器（例如 HiGHS 的 WASM 构建或等价方案），核对许可证、包体、确定性和 Worker 支持；不要手写未经充分验证的启发式算法替代精确求解器。

虚空场景三种模式的每单元黄金数量为：

| 模式 | 复制方向 | 主材料 | 隔离材料 | 设备 |
| --- | --- | ---: | ---: | ---: |
| 不堆积 | 无 | 110 | 14 | 1 |
| 平面 | X/Z | 93 | 31 | 1 |
| 立体 | X/Y/Z | 90 | 34 | 1 |

不堆积模式的一组黄金布局为：

```text
总尺寸：5 × 5 × 5
设备：(2, 2, 2)
最少隔离材料：14
主材料：110
隔离材料：14
设备：1

一组可行的隔离材料坐标：
(2,0,2),
(1,1,1), (3,1,1), (1,1,3), (3,1,3),
(2,2,0), (0,2,2), (4,2,2), (2,2,4),
(1,3,1), (3,3,1), (1,3,3), (3,3,3),
(2,4,2)
```

三个虚空最小值均已由原生 HiGHS 0-1 MILP 的相同模型离线证明上下界一致，最优坐标随版本化场景数据发布。

钚场景当前离线整数求解结果为：

| 模式 | 复制方向 | 钚块 | 铅块 | 集热器 | 已证明铅块下界 | 状态 |
| --- | --- | ---: | ---: | ---: | ---: | --- |
| 不堆积 | 无 | 88 | 36 | 1 | 33 | 可行 |
| 平面 | X/Z | 86 | 38 | 1 | 35 | 可行 |
| 立体 | X/Y/Z | 90 | 34 | 1 | 33 | 可行 |

这些钚布局已逐格验证，但上下界尚未一致，不能标记为最密或最优。Worker 构造预计算单元后仍按对应场景规则和周期边界独立复核；跨游戏版本使用前必须重新证明并更新规则数据。

### 7.2 求解流程

1. 将请求规范化为一个 `5 × 5 × 5` 单元，并生成固定设备坐标。
2. 按模式建立无回绕、X/Z 回绕或 X/Y/Z 回绕的稀疏约束矩阵。
3. 若场景数据包含当前模式的预计算坐标，则按其最优性状态直接构造单元；否则使用已知界限和 HiGHS 求解。
4. 在 Web Worker 中构造或求解，支持取消和请求序号，防止旧结果覆盖新输入。
5. 只接受整数解；按固定坐标顺序做确定性同分处理。
6. 将求解结果交给独立验证器，逐格检查设备位置、方块总数和对应场景的衰变约束。
7. 验证通过后生成统计、3D 实例数据和分层索引；验证失败则显示内部错误，不渲染错误结构。

### 7.3 性能边界

- 三种固定单元模型均在 Worker 中求解，主线程保持可交互；连续模式切换只保留最新请求。
- 已完成的请求可按规范化参数缓存，但缓存必须包含规则/数据版本。
- 3D 渲染按模型元素和材质批量实例化；分层视图只渲染当前二维切片。
- 性能验收同时关注求解耗时、峰值内存、首帧时间和相机交互帧率。

## 8. Minecraft 资源与 Three.js

现有资源位于 `src/assets/anvilcraft/`：

- 方块：`void_matter_block`、`negative_matter_block`、`plutonium_block`、`lead_block`。
- 设备：`void_energy_collector_base/head`、`heat_collector_base/head`。
- 模型是 Minecraft/Blockbench JSON，设备由 base 与 head 两部分组合；纹理是像素 PNG。

渲染适配层需要：

- 使用结构化 JSON 解析模型，不用字符串替换。
- 解析当前资源实际使用的 `parent`、`textures`、`elements`、`from/to`、逐面 UV、UV rotation、element rotation、`shade`、`light_emission`、`ambientocclusion` 和 `render_type` 子集。
- 内置解析 `minecraft:block/block` 与 `minecraft:block/cube_all`，遇到不支持的父模型或字段时明确失败。
- 正确处理反向 `from/to` 的内部面、透明/裁切纹理、发光轮廓和 base/head 的局部原点，不能把所有模型退化成同色立方体。
- 纹理使用最近邻采样、正确色彩空间和 alpha 规则，保持 Minecraft 像素观感。
- 缓存几何体、纹理和材质，并在组件卸载或场景重建时释放 GPU 资源。

模型和纹理的版权/许可证遵循仓库中的 `ASSETS_LICENSE`，修改、增删资源时同步更新来源说明。

- 所有求解状态、图例和错误不能只靠颜色表达。

## 10. 测试策略

### 10.1 领域与求解器

- 三种模式均规范化为 `1 × 1 × 1` 单元和 `5 × 5 × 5` 方块尺寸，忽略外部历史数量字段。
- 周期映射：不堆积不回绕，平面仅 X/Z 回绕，立体 X/Y/Z 回绕。
- 设备坐标、设备数量、坐标线性化/反线性化。
- 虚空场景中棱角边界主材料安全；单面边界或设备邻接的主材料仍有五个同类邻面时必须失败。
- 钚场景覆盖相邻钚块数减铅块数恰好为 `2` 时通过、为 `3` 时失败，并验证结构外和集热器不参与差值。
- 总位置守恒：`primary + separator + device = blocksX * blocksY * blocksZ`。
- 虚空三种模式黄金数量分别为 `110/14/1`、`93/31/1`、`90/34/1`，并验证独立布局和周期复制后的边界。
- 钚三种模式验证当前可行数量、已证明下界、独立布局和周期复制后的边界；上下界未一致时状态必须为 `feasible`。
- 对小网格使用穷举参考求解器，与生产 MILP 结果比较隔离材料最小值。
- 相同请求重复运行产生相同的隔离材料坐标。
- 超时可行解不得被标记为最优，验证失败的解不得进入视图。
- 周期边界用例必须覆盖单元两侧的回绕邻接，防止错误地把复制边界视为空气。

### 10.2 模型与渲染

- 为模型 JSON 解析器建立 fixture 测试，覆盖 `cube_all`、复杂 elements、反向盒、UV rotation、透明和发光字段。
- 验证所有场景资源路径存在、纹理可加载，base/head 能组成完整设备。
- 3D 与分层预览对同一坐标的方块类型一致。

- 使用 canvas 像素检查确认场景非空，并在桌面/移动视口截图验证自动取景、无重叠和内部裁切。

### 10.3 组件与端到端

- 场景切换、不堆积/平面/立体切换、复制方向、无数量输入、计算取消和结果更新。
- 3D 旋转/缩放/重置、角色显隐、裁切范围。

## 12. MVP 验收标准

- 用户可以选择虚空发电或钚块集热，并选择不堆积、平面或立体模式。
- 三种模式都生成一个 `5 × 5 × 5` 单元和一台中心设备，不要求用户输入堆积数量。
- 平面单元可沿 X/Z 复制，立体单元可沿 X/Y/Z 复制，复制后的每个单元布局完全一致。
- 每个主材料格都通过所属场景的衰变规则验证，不会出现已知衰变布局。
- 隔离材料数量在支持范围内被证明最少；未证明最优时界面不声称“最密”。
- 统计、3D 场景、分层网格和复制坐标来自同一份结果，逐格一致。
- 3D 预览使用仓库中的真实模型/纹理，可旋转、缩放、显隐和检查内部层。
- 分层预览支持三个轴、完整层范围、当前层统计和设备辨识。
- 三种模式求解均不阻塞主线程，3D 操作保持可用。
- 360 至 1440 px 宽度下核心流程无重叠、溢出或不可操作状态。
- `pnpm build`、测试和类型检查全部通过。
